# Прототип: Анализатор симптомов для записи к врачу

## 1. Ключевая гипотеза

Если мы предложим сервис маршрутизации по симптомам (продукт A) для людей 25-40 лет, которые ищут в интернете информацию о своих симптомах перед записью к врачу (сегмент B), в сценарии бытового ухудшения самочувствия (сценарий C), то мы получим конверсию в повторное использование не ниже 30% в течение 2 недель после первого использования (D >= 30%, T = 14 дней), потому что пользователи ценят сокращение времени на принятие решения о записи к нужному специалисту и снижение тревожности от неопределенности.

## 2. Ключевые метрики

### Продуктовые метрики

| Метрика | Значение | Способ измерения | Baseline | Частота |
|---------|----------|------------------|----------|---------|
| Доля пользователей, вернувшихся в бота повторно в течение 14 дней | >= 30% | Логи бота: подсчет уникальных user_id, совершивших >= 2 сессий за 14 дней от первой | 0% (продукта нет) | Непрерывно, отчет раз в 3 дня |
| Доля сессий, доведенных до конца (получен ответ с рекомендацией) | >= 60% | Отношение завершенных сессий к общему числу начатых | 0% | Непрерывно |
| Среднее время от начала ввода до получения ответа | <= 45 секунд | Логирование timestamp начала и конца сессии | Нет аналога | Раз в 3 дня |
| NPS (опрос после 3-го использования) | >= 30 | Кнопки в боте: "Насколько вероятно, что порекомендуете? 1-10" | Нет | Раз в неделю |

### Технические метрики

| Метрика | Значение | Способ измерения | Baseline | Частота |
|---------|----------|------------------|----------|---------|
| Uptime микросервиса | >= 99% | Healthcheck эндпоинт, опрос каждые 5 минут | Нет | Непрерывно |
| Время ответа API (p95) | <= 2 секунды | Логирование времени обработки запроса в FastAPI | Нет | Раз в день |
| F1-macro модели на новых данных | >= 0.55 | Еженедельная разметка 20 случайных запросов вручную | 0.05 (случайное угадывание 1/20) | Раз в неделю |

## 3. Скоуп прототипа

### Что войдет в прототип

- Прием текстового описания симптомов через Telegram-бота
- Классификация по 15-20 медицинским специальностям
- Ответ с указанием рекомендуемого специалиста и уровня срочности (планово / срочно)
- Готовый дисклеймер в каждом ответе
- Логирование всех запросов и ответов для анализа метрик

### Что намеренно не делаем и почему

- Уточняющие вопросы: увеличивают время сессии выше 45 секунд, усложняют архитектуру до диалоговой системы. Отложено до валидации базовой гипотезы.
- Голосовой ввод: требует интеграции Whisper, увеличивает latency и стоимость инференса. Не критично для проверки гипотезы.
- Геопривязка к поликлиникам: требует внешнего API и базы медучреждений. Не влияет на ключевую ценность.
- История обращений пользователя: требует авторизацию и хранение персональных медицинских данных, что влечет юридические риски без необходимости.
- Анализ загруженных медицинских документов: требует OCR и мультимодальных моделей.
- Монетизация через подписку или рекламу.

## 4. Готовые блоки и собственная разработка

### Берем готовое

| Компонент | Инструмент | Причина |
|-----------|------------|---------|
| NLP-модель для эмбеддингов | intfloat/multilingual-e5-small | Предобученная, показывает хорошее качество на русском, легковесная |
| Классификатор | CatBoost | Хорошо работает с табличными + эмбеддинговыми фичами, быстрый инференс |
| Telegram Bot API | python-telegram-bot или aiogram | Готовый фреймворк |
| Веб-сервер | FastAPI | Стандарт для ML-микросервисов, асинхронный, хорошая документация |
| Контейнеризация | Docker | Изоляция, воспроизводимость, простой деплой |
| Мониторинг | Логи в JSON + простой скрипт агрегации | Достаточно для MVP, не тянем Grafana/Prometheus |

### Делаем сами и почему это критично

| Компонент | Причина |
|-----------|---------|
| Датасет симптом-специалист | Готового открытого датасета на русском языке с разметкой "жалоба -> специалист" не существует. Это ядро продукта, качество здесь определяет всё |
| Пайплайн предобработки текста | Медицинский текст имеет специфику: разговорные формулировки, опечатки, бытовые названия болезней. Универсальные пайплайны не подойдут |
| Логика формирования итогового ответа | Ответ должен включать не просто метку класса, а дисклеймер, уровень срочности и обоснование. Это уникальная бизнес-логика |
| Шаблоны ответов и дисклеймер | Юридически значимый текст, не может быть сгенерирован автоматически, требует ручной проработки с учетом ограничений |

## 5. Главный риск прототипа и конечный формат

### Какой главный риск проверяем

Главный риск: пользователи не считают проблему выбора врача достаточно болезненной, чтобы использовать отдельный инструмент вместо поисковика. Мы проверяем, существует ли вообще потребность в быстрой маршрутизации, или люди предпочитают гуглить и читать несколько источников, несмотря на время и тревожность.

Второй риск: без эксперта домена точность модели может оказаться неприемлемо низкой для реального использования. Мы проверяем, можно ли достичь порогового качества (Top-1 Accuracy > 0.60) силами соло-разработчика с открытыми источниками и ручной разметкой.
